# .GITCLAW Help Lens

### A translation layer from CLI help documentation to the GitHub Issues UX

---

## What This Is

OpenClaw's existing help documentation (`docs/cli/`, `docs/help/`) is written for the terminal ‚Äî `openclaw <command> [options]`. When the agent lives inside a GitHub repository via `.GITOPENCLAW` (or `.GITCLAW`), the UX surface is **GitHub Issues**, not a shell. Users type comments, not commands.

This document is the **help lens**: a wrapper that maps every CLI help page to its GitHub Issues equivalent, with concrete examples a user can copy-paste into an issue comment and get a useful response.

The lens serves three purposes:

1. **For users** ‚Äî "I read the CLI docs; how do I do the same thing in an issue?"
2. **For the agent** ‚Äî a reference the agent can consult to translate `/command` requests into the right internal operations
3. **For contributors** ‚Äî a checklist of what is covered and what still needs implementation in the GitHub Issues UX

---

## Design Principles

| Principle | Rationale |
|---|---|
| **Natural language first** | Issue comments are freeform text. Users should be able to ask in plain English and get results. `/commands` are shortcuts, not requirements. |
| **`/command` as accelerator** | Structured `/command arg` syntax provides precision when the user wants it. The agent recognizes both natural language and `/command` prefixes. |
| **Markdown output** | Every response renders as a GitHub-flavored Markdown comment: tables, code blocks, collapsible sections, task lists, and linked references. |
| **Progressive disclosure** | Simple asks get simple answers. Append `--verbose` or ask "show me details" for the deep dive. |
| **Commit as side effect** | Commands that mutate state (config, memory, sessions) commit changes to `.GITOPENCLAW/state/` and link the commit in the response. |

## Detailed Command Mapping

Each section below maps a CLI help page (`docs/cli/<command>.md`) to its GitHub Issues equivalent with worked examples.

### `agent` ‚Üí Direct conversation

**CLI:** `openclaw agent --message "summarize the logs" --thinking medium`

**Issue comment:**
> Summarize the logs

Or with explicit control:
> `/agent --thinking medium summarize the logs`

Every issue comment **is** an agent turn. The agent reads the comment, thinks, and replies. No `/agent` prefix is needed for the common case ‚Äî just write your message.

For multi-turn conversations, keep commenting on the same issue. The agent resumes the session automatically via the issue-to-session mapping in `state/issues/<n>.json`.

### `gateway` ‚Üí Not directly applicable

The Gateway (`openclaw gateway run`) is a persistent WebSocket server. In the `.GITCLAW` model, GitHub Actions **is** the compute layer ‚Äî there is no long-running gateway process. Gateway-specific commands (`gateway start`, `gateway run`, `gateway status`) do not have direct Issue equivalents.

What the Gateway provides is decomposed into:
- **Agent execution** ‚Üí each issue comment triggers a GitHub Actions workflow
- **Channel routing** ‚Üí `/send` commands for outbound messages
- **Session persistence** ‚Üí git-committed JSONL files
- **Health monitoring** ‚Üí `/status` and `/doctor`

## Natural Language Examples

Users do not need to memorize `/commands`. The agent understands natural language requests and maps them to the appropriate action:

| Natural language request | What the agent does |
|---|---|
| "What model are you using?" | Reads `config/settings.json` and replies with the model name |
| "Switch to GPT-4o" | Runs `/models set gpt-4o` internally |
| "Show me recent conversations" | Runs `/sessions list` internally |
| "Search your memory for authentication bugs" | Runs `/memory search "authentication bugs"` internally |
| "Run diagnostics" | Runs `/doctor` internally |
| "What's your status?" | Runs `/status` internally |
| "Tell the team on Slack that the release is out" | Runs `/send slack #general "the release is out"` internally |
| "What skills do you have?" | Runs `/skills list` internally |
| "Create a daily job to check for stale PRs" | Runs `/cron create "stale PR check" "0 9 * * *" "List PRs with no activity in 7 days"` internally |
| "Are there any updates available?" | Runs `/update status` internally |
| "Change your thinking level to low" | Runs `/config set defaultThinkingLevel low` internally |

## What Does Not Translate

Some CLI commands are inherently local or require persistent infrastructure that GitHub Actions cannot provide:

| CLI Command | Why it does not translate | Alternative |
|---|---|---|
| `openclaw gateway run` | Requires a persistent process | GitHub Actions is the compute; no gateway needed |
| `openclaw gateway start` | Same as above | N/A |
| `openclaw tui` | Terminal UI requires a TTY | Use the issue thread as the conversational interface |
| `openclaw browser` | Requires a headed browser session | Agent has access to web tools within the Actions runner |
| `openclaw daemon` | System service management | GitHub Actions manages lifecycle |
| `openclaw qr` | Generates terminal QR codes for pairing | Not applicable; no local device pairing in Issues UX |
| `openclaw pairing` | Device-to-device pairing | Not applicable in the Issues context |
| `openclaw nodes` / `openclaw devices` | Multi-device mesh management | Single-node execution on Actions runner |
| `openclaw configure` | Interactive terminal wizard | Use `/config set` commands or open a "Setup" issue for guided onboarding |
| `openclaw sandbox` | Container sandbox management | Actions runner is the sandbox |
| `openclaw reset` | Destructive state reset | Ask the agent: "Reset your state" ‚Äî scoped to `.GITOPENCLAW/state/` |
| `openclaw uninstall` | Remove OpenClaw from the system | Delete the `.GITOPENCLAW/` folder and its workflow |

## Response Format Guidelines

When the agent responds to `/commands` in the Issues UX, the output should follow these formatting conventions to match what users expect from the CLI but adapted for Markdown:

| CLI Output | Issue Comment Output |
|---|---|
| ANSI-colored table | Markdown table |
| Progress spinner | üëÄ reaction during processing ‚Üí ‚úÖ on completion |
| Code block output | Fenced code block with language hint |
| Error with exit code | ‚ùå callout block with error description and remediation |
| Interactive prompt | Follow-up question in a comment; user replies to continue |
| JSON output (`--json`) | Fenced JSON code block with collapsible `<details>` wrapper for large payloads |
| Streaming response | Single complete comment (no partial updates) |
| File output | Committed to repo with a link to the commit |
| Hyperlinks (OSC-8) | Standard Markdown links |

## Implementation Notes

For contributors implementing `/command` support in the agent orchestrator:

1. **Command parsing**: The orchestrator (`GITOPENCLAW-AGENT.ts`) receives the full comment body. A prefix parser detects `/command` patterns and extracts the command name and arguments. See the equivalence map in `.ANALYSIS-OPENCLAW-FUNCTIONALITY-EQUIVALENTS.md` ¬ß2.2 for the canonical mapping.

2. **Fallback to natural language**: If no `/command` prefix is detected, the full comment is passed to the agent as a natural language message. The agent's system prompt includes awareness of these commands and can invoke them internally.

3. **State mutations are commits**: Any command that changes configuration or state must `git add .GITOPENCLAW/` ‚Üí `git commit` ‚Üí `git push` before replying. The commit SHA should be included in the response.

4. **Secrets are never in comments**: Commands requiring credentials (channel tokens, API keys) must redirect the user to GitHub's **Repository Settings ‚Üí Secrets** UI. The agent must never ask for or accept secrets in issue comments.

5. **Output truncation**: GitHub has a ~65,536 character limit on issue comments. Large outputs (session dumps, memory searches with many results) should use `<details>` collapsible sections or pagination ("showing 10 of 47 results ‚Äî reply `/more` for the next page").

6. **Help command**: `/help` or "What can you do?" should return a compact version of the Quick Reference Card above, tailored to the commands actually implemented.
